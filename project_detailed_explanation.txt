====================================================
SECTION 1: WHAT IS THIS PROJECT?
====================================================

NSE Trading Journal is a personal stock trade tracking
web application built for Indian equity traders.

It allows a trader to:
  - Record every buy/sell trade they make
  - Automatically calculate profit/loss and % return
  - See a performance dashboard with summary stats
  - Visualize P&L through bar and line charts
  - Get AI-style insights on individual trades
  - Get an overall portfolio performance summary
  - Search and autocomplete Indian stock names

The entire app runs locally on your machine — no
internet connection required after setup (except for
loading Chart.js from CDN on first use).

====================================================
SECTION 3: TECHNOLOGY STACK — DETAILED
====================================================

--- BACKEND: Flask (Python) ---

Flask is a lightweight Python web framework.
It handles:
  - URL routing (which function runs for which URL)
  - Form data processing (request.form)
  - Query parameter reading (request.args)
  - HTML rendering via Jinja2 templates
  - JSON responses for AJAX calls
  - Redirect responses after form submissions

Key Flask concepts used:
  - @app.route()       : maps URL to function
  - render_template()  : renders HTML with data
  - redirect()         : sends browser to another URL
  - url_for()          : generates URL from function name
  - request.form       : reads POST form data
  - request.args       : reads GET query params
  - jsonify()          : returns JSON response
  - g object           : per-request global storage
  - @app.teardown_appcontext : cleanup after request

--- DATABASE: SQLite ---

SQLite is a file-based relational database.
No server needed — just a .db file on disk.

The "trades" table schema:
  id           INTEGER  PRIMARY KEY AUTOINCREMENT
  stock_name   TEXT     NOT NULL
  buy_price    REAL     NOT NULL
  sell_price   REAL     NOT NULL
  quantity     REAL     NOT NULL
  buy_date     TEXT     NOT NULL  (format: YYYY-MM-DD)
  sell_date    TEXT     NOT NULL  (format: YYYY-MM-DD)
  holding_days INTEGER
  profit       REAL
  pct_return   REAL

Parameterized queries (?) are used throughout to
prevent SQL injection attacks.

--- FRONTEND: HTML + CSS + JavaScript ---

The entire UI is a single HTML file (index.html).
No frontend framework (no React, Vue, Angular).

HTML structure:
  - <section> for Performance Overview (summary cards)
  - <section> for P&L Chart (Chart.js canvas)
  - <section> for Add Trade form
  - <section> for Trade History table

CSS highlights:
  - Dark theme: background #0f0f1a (deep navy-black)
  - Glassmorphism-style cards with gradient borders
  - Responsive flex layout (wraps on small screens)
  - Smooth hover transitions on cards and buttons
  - Color coding: green (#34d399) = profit,
                  red (#f87171) = loss,
                  purple (#818cf8) = neutral/AI

JavaScript features:
  - Autocomplete: fetch('/suggest?q=...') on input
  - Inline edit toggle: show/hide edit row per trade
  - AI Insight: fetch('/ai/insight/<id>') on click
  - AI Summary: fetch('/ai/summary') on click
  - Chart.js: bar/line chart with tab switching

--- CHART LIBRARY: Chart.js ---

Loaded from CDN:
  https://cdn.jsdelivr.net/npm/chart.js

Used to render a P&L bar/line chart.
Data is injected from Flask via Jinja2:
  rawLabels  = list of stock names
  rawProfits = list of profit values

Colors are dynamically assigned:
  green for profit trades, red for loss trades.


====================================================
SECTION 4: BACKEND FUNCTIONS — DETAILED
====================================================

--- get_db() ---
Purpose: Returns a SQLite connection for the current
         HTTP request. Reuses existing connection if
         already opened (stored in Flask's g object).
Called by: All route functions that need DB access.

--- init_db() ---
Purpose: Creates the trades table on first run.
         Also migrates old databases by adding missing
         columns (buy_date, sell_date, holding_days).
Called by: app startup (if __name__ == '__main__')

--- calculate_pnl(buy, sell, qty) ---
Purpose: Computes profit and % return for a trade.
Formula:
  profit     = (sell_price - buy_price) * quantity
  pct_return = (profit / (buy_price * quantity)) * 100
Returns: tuple (profit, pct_return)

--- holding_days(buy_date_str, sell_date_str) ---
Purpose: Calculates number of days between buy and
         sell date.
Method:  Parses both strings as datetime objects,
         subtracts, returns max(0, days).
Returns: integer (0 if dates are same or invalid)

--- get_all_trades() ---
Purpose: Fetches all trades from DB, newest first.
Query:   SELECT * FROM trades ORDER BY sell_date DESC
Returns: list of dicts (one per trade)

--- get_summary(trades) ---
Purpose: Computes aggregate stats from trade list.
Returns: dict with keys:
  total_profit, total_loss,
  win_count, loss_count, win_loss_ratio


====================================================
SECTION 5: ROUTES — DETAILED
====================================================

GET  /
  - Loads all trades and summary
  - Renders index.html with full data
  - Passes optional message/error from query params

POST /add
  - Reads 6 form fields
  - Validates: presence, numeric types, positivity,
    date order
  - Calculates P&L and holding days
  - Inserts into DB
  - Redirects to / with success message

POST /edit/<id>
  - Same validation as /add
  - Updates existing trade row by id
  - Redirects to / with success message

POST /delete/<id>
  - Deletes trade by id
  - Redirects to /

GET  /suggest?q=<query>
  - Returns JSON array of up to 8 stock name matches
  - DB results (user's own stocks) come first
  - KNOWN_STOCKS list fills remaining slots

GET  /ai/insight/<id>
  - Fetches single trade from DB
  - Runs rule-based analysis on pct_return,
    holding_days, and price move volatility
  - Returns JSON: { "insight": "..." }

GET  /ai/summary
  - Fetches all trades
  - Runs rule-based analysis on win/loss ratio,
    net P&L, avg holding days, best/worst trade
  - Returns JSON: { "summary": "..." }


====================================================
SECTION 6: FRONTEND INTERACTIONS — DETAILED
====================================================

--- AUTOCOMPLETE ---
Trigger: User types in the "Stock Name" input field
Flow:
  1. JavaScript listens to 'input' event
  2. Sends GET /suggest?q=<typed text>
  3. Flask returns JSON array of matches
  4. JS renders dropdown list below the input
  5. User clicks a suggestion → fills the input
  6. Clicking outside → hides the dropdown

--- INLINE EDIT ---
Trigger: User clicks "Edit" button on a trade row
Flow:
  1. toggleEdit(id) is called
  2. The view row (id="view-X") fades to 50% opacity
  3. The edit row (id="edit-X") appears below it
  4. Edit row contains a pre-filled form
  5. User edits fields and clicks "Save"
  6. Form POSTs to /edit/<id>
  7. "Cancel" hides the edit row and restores opacity

--- AI INSIGHT (per trade) ---
Trigger: User clicks "🤖 Insight" button on a trade
Flow:
  1. loadInsight(id, btn) is called
  2. Button shows "⏳" and is disabled
  3. Sends GET /ai/insight/<id>
  4. Flask returns JSON with insight text
  5. Text is shown in a blue box below the buttons
  6. Clicking again toggles the box off

--- AI SUMMARY (overall) ---
Trigger: User clicks "🤖 AI Performance Summary"
Flow:
  1. loadAISummary() is called
  2. Box shows "⏳ Analyzing your trades..."
  3. Sends GET /ai/summary
  4. Flask returns JSON with summary text
  5. Text replaces the loading message
  6. Clicking again toggles the box off

--- P&L CHART ---
Trigger: Page load (if trades exist)
Flow:
  1. Jinja2 injects trade data as JSON into JS vars
  2. buildChart('bar') is called on page load
  3. Chart.js renders a bar chart
  4. Green bars = profit trades
  5. Red bars = loss trades
  6. "Bar" / "Line" tabs switch chart type
  7. showChart() destroys old chart and rebuilds


====================================================
SECTION 7: DATA FLOW — END TO END
====================================================

Example: User adds a trade for RELIANCE

  1. User fills form:
       Stock: RELIANCE
       Buy: 2400, Sell: 2600, Qty: 10
       Buy Date: 2024-01-01, Sell Date: 2024-01-15

  2. Browser POSTs to /add

  3. Flask reads form data

  4. Validation passes (all fields valid, sell > buy date)

  5. calculate_pnl(2400, 2600, 10):
       profit     = (2600 - 2400) * 10 = ₹2000
       pct_return = (2000 / 24000) * 100 = 8.33%

  6. holding_days("2024-01-01", "2024-01-15"):
       days = 14

  7. INSERT INTO trades:
       (RELIANCE, 2400, 2600, 10,
        2024-01-01, 2024-01-15, 14, 2000, 8.33)

  8. Redirect to / with message "Trade added. Held 14 days"

  9. index() loads all trades + summary, renders page

  10. User sees RELIANCE row in green with +₹2000 / +8.33%


====================================================
SECTION 8: SECURITY CONSIDERATIONS
====================================================

GOOD:
  - Parameterized SQL queries prevent SQL injection
  - Input validation rejects empty, non-numeric,
    negative, and date-order-invalid inputs
  - No external API keys or secrets in code

LIMITATIONS:
  - No CSRF protection on forms
  - No authentication (anyone on the machine can access)
  - Stock names are rendered directly in HTML
    (potential XSS if malicious names are entered)
  - Debug mode is ON (app.run(debug=True)) —
    should be turned OFF in any production deployment


====================================================
SECTION 9: KNOWN DESIGN DECISIONS
====================================================

1. WHY SQLITE?
   Simple, zero-config, file-based. Perfect for a
   single-user local app. No need to install MySQL
   or PostgreSQL.

2. WHY NO FRONTEND FRAMEWORK?
   Keeps the project dependency-free on the frontend.
   Vanilla JS is sufficient for the interactivity needed.

3. WHY RULE-BASED AI?
   No API key needed, works offline, fully transparent.
   The "AI" is just well-structured if/elif logic that
   reads like a trading coach's advice.

4. WHY JINJA2 FOR CHART DATA?
   Avoids a separate API call for chart data on page
   load. Trade data is already available server-side,
   so it's injected directly into the JS variables.

5. WHY REDIRECT AFTER POST?
   Follows the POST-Redirect-GET pattern. Prevents
   duplicate form submissions if the user refreshes
   the page after adding/editing a trade.
